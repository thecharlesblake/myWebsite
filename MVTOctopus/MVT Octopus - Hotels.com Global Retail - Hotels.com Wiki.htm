    MVT Octopus

    Skip to end of metadata
    • Created by Charlie Blake, last modified just a moment ago
    Go to start of metadata

     

     

     

     

     

     


     

     

     

     

     

     


    Background

    The aim of creating MVT Octopus was to improve the reliability and speed at which Site Ops could launch MVTs. The basic process involves a number of systems (see MVT Launching Procedures) which, prior to Octopus did not "talk to each other". These include YARP, the "MVT Changes" Google Sheet, JIRA, SiteSpect, Athena and Terastorm (these are the "tentacles" of Octopus).

    Information about the requested changes from the Google Sheet, along with the corresponding change details from SiteSpect / Athena, must first be put into a JIRA comment. Then the status of the JIRA ticket may then need to be updated, and the same information could also then be required in TeraStorm. Manually copying and pasting all of this information from one application to another was both slow and error prone. Thus, it was decided that an application should be created to automate as much of the MVT launching procedure as possible. This is the problem that MVT Octopus was created to solve.

    Setup

    To install Octopus you will need access to the SiteOps dropbox. If you don't have access, please speak to Andrea Mestriner (amestriner@hotels.com). Within the SiteOps folder, go to the "MVT Octopus" directory.

    From here, the setup instructions are all included in the README file which you'll find there, but for the sake of ease the contents of that file are as follows:

    There are two parts of Octopus which users will need to install: the SEA app and the Jumpbox app.

    SEA App Installation

    Before you do anything, please copy this directory from dropbox into a local area in your filesystem. This is just to avoid any problems during the install.

    (Note: part of both steps involves running a "setup" script. This setup script adds the HCom JIRA certificate to your Java keystore. The reason for this is due to an issue with our JIRA SSL certificate, which means that Java doesn't inherently trust it. You don't really need to worry about this as a user of Octopus though - I just mention it here for the record).

    Mac Installation

    1. Run the "setup-mac" file. Frustratingly, Dropbox changes the user permissions on the file when it's uploaded, so you'll need to make it executable yourself before you can actually run it. To do this, open up the terminal (search for "terminal" on Mac), type

      cd "Downloads/MVT Octopus" (replace [Downloads] with wherever you installed the folder. The quotes must be included!)

      and press Enter. Then type

      chmod 755 setup-mac

      and press enter again. Now you have permissions to run the setup script, which you can do by entering

      ./setup-mac

      The message "Certificate was added to keystore" should be outputted. Then you're ready to go!

    2. If at the final command above you receive a message saying that the command failed because of a JDK issue, you will need to go to http://www.oracle.com/technetwork/java/javase/downloads/index.html and install the latest JDK. Then run the command again.
    3. Now, to install the actual program, simply double click the .dmg file and follow the instructions onscreen.
    4. You may at some stage get a message which says "MVT Octopus can’t be opened because it is from an unidentified developer" which stops you from starting Octopus. To fix this, ctrl-click the Octopus icon and select "Open". It will then give you the option of telling the system that you trust the program.

     

    Windows Installation

    1. Run the "setup-windows.bat" file. To do this, simply double click the file and follow the onscreen instructions.
    2. Now, to run the app, simply double click "MVT_Octopus.jar". For ease, you may wish to right click and create a shortcut to the program on your desktop / taskbar.

     

    Jumpbox App Installation

    1. To install the jumpbox section of the app, you will first need understand the shared filesystem between SEA and the Jumpbox. See Transferring Files in Production for more information.
    2. Go to "\\che-filidx.idx.expedmz.com\NonPCITransfer\ToCorp\Public\MVTAssist" in the Jumpbox, and drag "MVT Octopus - Jumpbox.crx" to your desktop.
    3. Open chrome and type "chrome://extensions/" into the search bar.
    4. Drag the .crx file from your desktop into the chrome extensions page. A message should pop up allowing you to install the app.
    5. Navigate to "chrome://apps/" and right click on the Octopus icon, selecting "Create shortcuts...".
    6. Now you have the Jumpbox app set up on your machine. When you open it some setup settings will need to be filled in (follow the onscreen instructions). These must be entered the first time you use the app, but will be remembered for subsequent sessions.

    Features

    Link to Google Sheet

    When Octopus starts up, the first thing that takes place is a data pull from the "MVT Changes" Google Sheet. The link to this sheet is as follows: https://docs.google.com/spreadsheets/d/1G_k62FwtSmCWQoRsyyh8cbsDHGDtRd023slvmufDgRg. If you don't have access to this document, get in touch with Annaelle Poisson (apoisson@hotels.com).

    This data is pulled into the main Octopus table. To sync the sheet with Octopus the user is provided with two buttons: "Cloud Pull" and "Cloud Push" respectively.

    Octopus will never sync with the cloud unless you explicitly tell it to do so. It will also do its best to merge changes in an "intuitive" way. For instance, if the user makes a change to cell B1 in Octopus, and then someone else makes a change to B2 in the Google Sheet, a "Cloud Pull" will update B2 in Octopus, keeping the user change made in B1. However, if both users had made different changes to the same cell, pressing "Cloud Pull" would fail and flag that the given cell has a "merge conflict". In this scenario, the user would have to manually change the cell in the Sheet and in Octopus to match, and the reattempt the pull.

    A "Cloud Push" will always in the background perform a "Cloud Pull" first, meaning you never have to worry about a push overwriting a change in the cloud. Note that on the far right of the Octopus table there are 5 "extra" columns which are not synced with the sheet. We will come to the purpose of these in the next section.

    Link To JIRA

    One of the most powerful features of Octopus is its capacity to auto-generate change comments for Jira. Depending on the contents of the "Test Type", "Test Tool" and "Change Type" columns for a particular row, Octopus will create a "template comment" for that row. The comment box at the bottom of the app will show the comment corresponding to the row which your selected cell is in.

    For instance, if the selected row has "Test Type", "Test Tool" and "Change Type" corresponding to "MVT", "Sitespect" and "Activate", the template comment will be as follows:

    [CRQ]
    [Change Date]

    Test activated
    [split ratio]

    HCOM
    Desktop & Tablet & Mobile Web
    All POSa
    [URL]

    SS IDs:
    Test:
    Control:
    Variation

    The three lines "HCOM, Desktop & Tablet & Mobile Web, All POSa", are extracted from the "Brands", "Devices", and "POSa" columns in the table. The [CRQ] line will be filled in when a CRQ has been added, as will the [Change Date] (see the Miscellaneous section about an easy way to add CRQs and change dates). In the next section we'll cover how to auto-populate the rest of the comment. Different types of request will have different templates, and if the request type is not recognised, a default template will be generated with some basic information.

    The comment box is editable, and if you make a change to it and then select another comment, that custom change will be remembered. However, if you wish to undo that change, you can always select the "Reset Comment" button, which changes the comment text back to the template. Sometimes, an update to a column such as the "Brand" or the "POSa" does not automatically propagate into a change in the comment. However, by pressing the "Reset Comment" button a new template will be generated with the updated details. Be aware though, that this will overwrite any custom changes made to the text.

    When a comment is ready to go simply click the "Send Comment" button and a success (or failure) dialog will appear. If you wish to send multiple comments at once, just select muliple rows and press the send button. If you then wish to see the comment on the JIRA ticket, just click the "Go to JIRA Ticket" button in the top-right corner.

    You can also change the status of a JIRA ticket from within Octopus by pressing the "Edit JIRA Status" Button. The resulting dialog should explain how exactly these state transitions can be made.

    Link To Jumpbox, Sitespect and Athena

    The Jumpbox section of Octopus is designed to provide the SEA app access to the Sitespect API (which can only be accessed in the Jumpbox). In brief, the way this works is by writing JSON to and from the shared SMB filesystem between the two environments. What this enables the user to do, is pull Sitespect URLs, Test IDs, and Variation / Control codes, which can then be used in both JIRA and Terastorm. All Sitespect tests need URLs, and this is especially useful for Sitespect "Activates", which also need the rest of the aforementioned info.

    To actually pull the data, select the required row(s), and press the "Jumpbox Pull" button in the SEA app. Then, inside your Jumpbox, click the Octopus icon. When the Jumpbox app opens it automatically runs, and pulls the relevant data (you should see it say "Complete" when it's finished). The "Run" button will let you repeat the process. Once this is done, back in the SEA app you should see a confirmation message. The pulled data will then be automatically filled in the final 5 columns of the table. In cases where there were multiple matching campaigns in Sitespect, sometimes at this stage a dialog may also appear requiring you to select the campaign you want.

    This will then enable the auto-filling of the rest of the JIRA comment. In the example in the previous section, this would fill in the [URL], Test ID, Control ID and Variation IDs. For Athena, there is currently no tool to pull data in an automated way. However, for "Activates", if you fill in the "Athena Variant Spread" column, in the body of the comment it will auto-generate the "variant-spread" line (this will also be useful later for Terastorm).

    NOTE: on a few occasions we have experienced problems with the syncing of the shared Jumpbox filesystem. Please be aware of this if issues arise with this part of the tool.

    Link to Terastorm

    For those not familiar with Terastorm, this is the tool used to register MVTs so that they appear in our Heart reporting. It can be accessed at http://terastorm. This section of Octopus is designed to replicate the "enter new test" form in Terastorm, auto-populating it with information from the table.

    The Terastorm part of Octopus is more limited than Terastorm itself, and when you click the "Terastorm!" button it gives you three options: "Activate", "Complete" and "Launch".

    "Complete" simply enables you to change a test from "Live" to "Complete" and set an end-date manually via a dialog. The dialog suggests the date in the "Change date" column, but you can set it to whatever you want.

    "Activate" allows you to create a new test from scratch (as shown above). It can populate all of the above fields from the sheet, as long as they are all filled in correctly. This includes the codes for each variation, which are created by parsing the text in the 5 "extra" columns in the table - make sure these are filled in correctly! Octopus also knows if previous versions of this MVT exist in Terastorm already, and updates the verison number accordingly. In 95% of cases you should be fine just selecting "OK" straight away.

    One important thing to note about "Activate" and "Complete" is that they do not send the request to Terastorm themselves. Instead Octopus remembers the requests and readies them to be launched at your discretion. This is where the third button, "Launch" comes in.

    This button will take you to a screen showing all of your stored Terastorm requests. You can then inspect them, remove ones you don't want to send, and finally, press the "Launch" button within this dialog to make all of the requests in parallel (note: doing this will launch all the requests in the table - not just the ones selected by the cursor). Why does Octopus work this way? Largely because the Terastorm tool is so slow that making the requests individually / sequentially can be quite painful. By storing up all your changes and making then in parallel the process is sped up greatly.

    Miscellaneous Features (including YARP)

    When the user has deployed a change in Sitespect or Athena, they should then tick the "deployed" checkbox for the relevant row. Note that this auto-generates a date in the "Change Date" column, which can then be used in JIRA and Terastorm.

    The "Add CRQ" button is designed to make adding YARP CRQs easier. Simply select cells from the rows whose CRQ column you wish to update, click the button, and enter the CRQ text.

    If the user wishes to copy cells from the table, they can use ctrl- and shift- click to select multiple items at once. Once they are selected, pressing the "Copy" button will then copy the text of the selected cells to the clipboard. This is particularly useful for getting a list of MVTs to enter into YARP.

    Technical Details

    MVT Octopus was built by Charlie Blake (SiteOps intern, Summer 2016). The main SEA app was built as a JavaFX desktop application, and the Jumpbox part of the tool was built as a Javascript Google App.

    Note that during the production of Octopus it was known as "MVT Assist" or "MVT Automation" - these names are still used in certain internal parts of the tool. The source for the program can be found in the SiteOps section of Stash in the "MVTAutomation" directory. To build this tool from source, run the "build.xml" file in the "build/" directory.
